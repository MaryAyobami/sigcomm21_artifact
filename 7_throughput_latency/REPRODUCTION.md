# Throughput/Latency Experiment Reproduction Guide

Reproduces the performance claim from section 7/8 of the K2 paper: K2-compiled XDP programs
achieve lower tail latency than clang-O2 at the same or better throughput (MLFFR).

Seven variants of `xdp_fwd` are benchmarked: `o1`, `o2` (clang), and `k0`–`k4` (K2
superoptimizer output). The K2 objects in `k2-objects/` were generated by the K2 compiler
and extracted from the original xl170 pre-baked artifact image.

| c6320 | Clemson | ixgbe | hardcode in profile | template |
| r6525 | Clemson | mlx5 | auto (T-Rex ARP) | template |
| sm220u | Wisconsin | mlx5 | auto (T-Rex ARP) | template |
| d7615 | Clemson | mlx5 | auto (T-Rex ARP) | template |

**MAC handling**: mlx5 nodes — T-Rex resolves node-0's MAC at startup via ARP, no
manual entry needed. ixgbe/i40e/ice nodes — T-Rex cannot ARP, so `udp_for_benchmarks_<type>.py`
must hardcode node-0's MAC and a static ARP entry must be set on node-0.

## Step 1 — Allocate a CloudLab experiment

Instantiate the profile from the `hardware-generalization` branch of this repo. Select
your target node type from the dropdown. Both nodes get the same hardware type.

Wait for the experiment to reach **Ready** state. For non-xl170/d6515 types, the boot
script installs T-Rex (~15 minutes). Get both hostnames from the List View.

For **c6620 only**: after the experiment is ready, SSH into both nodes and install the
HWE kernel before anything else:

```bash
sudo apt-get install -y linux-image-generic-hwe-20.04 && sudo reboot
```

Wait for both nodes to come back up (`uname -r` should show `5.15.x`), then continue.

## Step 2 — Gather hardware information

SSH into **node-1** (the traffic generator).

**Find the experimental NIC PCI address:**
```bash
lspci | grep -i ethernet
# Note the bus ID of the 10G/25G/100G NIC, e.g. "03:00.0" or "82:00.0"
```

**Get the NIC MAC** (must be done before the benchmark starts — DPDK hides the NIC after binding):
```bash
# Find the interface on the 10.10.1.2 address:
ip addr show | grep -B4 "10.10.1.2" | grep "link/ether"
# or from dmesg if the interface isn't visible:
dmesg | grep "<pci-addr>" | grep -oE "([0-9a-f]{2}:){5}[0-9a-f]{2}" | head -1
```

**Find the NUMA node of the NIC:**
```bash
cat /sys/bus/pci/devices/0000:<pci-addr>/numa_node
# e.g. cat /sys/bus/pci/devices/0000:03:00.0/numa_node
```

**Get CPU list for that NUMA node** (replace `1` with your NUMA node number):
```bash
lscpu -p=CPU,NODE | grep -v "^#" | awk -F, '$2==1 {print $1}' | tr '\n' ','
```
Pick one CPU for `master_thread_id`, a different one for `latency_thread_id`, and the
remaining CPUs on that socket go in `threads`.

**Get the interface name on node-1:**
```bash
ip addr show | grep "10.10.1.2" 
# the interface name is in the line above, e.g. "enp130s0f0" or "ens1f1"
```

**Get node-0's experimental NIC MAC** (needed for Intel NIC node types):
```bash
ssh <user>@<node-0-hostname> 'ip addr show | grep -B4 "10.10.1.1" | grep "link/ether"'
```

## Step 3 — Set up config files

Config files live in `7_throughput_latency/trex-configuration/` in the cloned repo at
`/tmp/sigcomm21_artifact` on both nodes (put there by the boot script).

### Validated node types (xl170, d6515, c8220, sm220u, d7615)

Config files already exist. For **ixgbe/i40e/ice nodes (c8220, c6320, c6420, c6620)**:
update the destination MAC in the traffic profile — it is hardcoded and changes with
every new experiment allocation:

```bash
# On node-1:
sed -i 's/dst="[0-9a-f:]*"/dst="<node-0-mac>"/' \
    /tmp/sigcomm21_artifact/7_throughput_latency/trex-configuration/udp_for_benchmarks_<type>.py
```

For **mlx5 nodes (xl170, d6515, sm220u, d7615)**: no MAC update needed — T-Rex
resolves node-0's MAC automatically via ARP at startup.

### New node types (c6525-25g, c6420, c6620, c6320, r6525)

Create three files by copying from the nearest validated type:

**a) T-Rex YAML config** — copy xl170 (single port) and fill in your hardware values:
```bash
cp /tmp/sigcomm21_artifact/7_throughput_latency/trex-configuration/trex_cfg_xl170.yaml \
   /tmp/sigcomm21_artifact/7_throughput_latency/trex-configuration/trex_cfg_<type>.yaml
```

Edit the new file:
```yaml
- version: 2
  interfaces: ["<pci-addr>", "dummy"]    # e.g. "03:00.0"
  port_info:
    - ip: 10.10.1.2
      default_gw: 10.10.1.1
    - ip: 10.10.2.2
      default_gw: 10.10.2.1
  platform:
    master_thread_id: <cpu-A>            # one CPU on the NIC's NUMA node
    latency_thread_id: <cpu-B>           # different CPU, same NUMA node
    dual_if:
      - socket: <numa-node>
        threads: [<remaining CPUs on that NUMA node>]
```

**b) Traffic profile** — for mlx5 nodes copy from xl170, for ixgbe/i40e/ice nodes copy
from c8220 and update the MAC:

```bash
# mlx5 (c6525-25g, r6525):
cp /tmp/sigcomm21_artifact/7_throughput_latency/trex-configuration/udp_for_benchmarks_xl170.py \
   /tmp/sigcomm21_artifact/7_throughput_latency/trex-configuration/udp_for_benchmarks_<type>.py

# ixgbe/i40e/ice (c6420, c6620, c6320):
cp /tmp/sigcomm21_artifact/7_throughput_latency/trex-configuration/udp_for_benchmarks_c8220.py \
   /tmp/sigcomm21_artifact/7_throughput_latency/trex-configuration/udp_for_benchmarks_<type>.py
sed -i 's/dst="[0-9a-f:]*"/dst="<node-0-mac>"/' \
    /tmp/sigcomm21_artifact/7_throughput_latency/trex-configuration/udp_for_benchmarks_<type>.py
```

**c) Interface name config** — one line, the kernel interface name on node-1:
```bash
echo "<interface-name>" > \
    /tmp/sigcomm21_artifact/7_throughput_latency/trex-configuration/scripts/<type>.config
# e.g. echo "enp3s0f0" > .../scripts/c6525-25g.config
```

## Step 4 — Set up node-0

SSH into **node-0**.

**Install BPF build dependencies:**
```bash
sudo apt-get install -y clang llvm libbpf-dev libelf-dev
```

**Compile XDP programs and install K2 objects:**
```bash
cd /tmp/sigcomm21_artifact/7_throughput_latency
./setup_xdp_node0.sh <node-1-nic-mac>
```

This compiles `xdp_fwd` at O1 and O2, installs the real K2 objects from `k2-objects/`
into the benchmark directory, and sets a static ARP entry for node-1's IP. It takes
2–5 minutes.

**Verify XDP loads in native mode (not generic):**
```bash
cd /usr/local/throughput-experiments
sudo ./xdp_fwd <node-0-interface>
ip -d link show <node-0-interface> | grep xdp
# Must show "xdp" — if it shows "xdpgeneric" the NIC does not support native XDP
sudo ip link set dev <node-0-interface> xdp off
```

**For Mellanox nodes (mlx5 driver)**: install MLNX_OFED matching the NIC generation
before running `setup_xdp_node0.sh`. ConnectX-4 (xl170): MLNX_OFED 5.0-2.1.8.0.
ConnectX-5 (c6525-25g, r6525, d6515): MLNX_OFED 5.4+. ConnectX-6 (sm220u, d7615):
MLNX_OFED 5.6+. Download from https://network.nvidia.com/products/infiniband-drivers/linux/mlnx_ofed/

## Step 5 — Set up node-1

SSH into **node-1**.

```bash
cd /tmp/sigcomm21_artifact/7_throughput_latency
./setup.sh <node-type> <user>@<node-0-hostname>
source ~/.bash_profile
```

Example:
```bash
./setup.sh c8220 Mary@hp084.utah.cloudlab.us
source ~/.bash_profile
```

This copies configs to `/usr/local/`, installs the fixed benchmark scripts, and sets
`PYTHONPATH` for the T-Rex Python API.

---

## Step 6 — Run the benchmark

On **node-1**:

```bash
cd /usr/local/v2.87

# Full run — all 7 variants, 3 iterations each (~8 hours):
nohup python3 -u run_mlffr.py -b xdp_fwd -d xdp_fwd_<type>_all -n 3 -c 6 \
    > ~/xdp_fwd_<type>_full.log 2>&1 &

# Monitor:
tail -f ~/xdp_fwd_<type>_full.log
```

**Arguments:**
- `-b xdp_fwd` — benchmark name
- `-d <dir>` — output directory (created under `~/`)
- `-n 3` — 3 iterations per variant
- `-c 6` — T-Rex cores (adjust down if the node has fewer than 8 cores on one NUMA node)
- `-v o2` — run a single variant (omit for all 7)
- `-mS 4.4 -mE 5.4 -i 0.1` — override the default Mpps search range if needed

**Quick single-variant sanity check** (~30 minutes):
```bash
python3 -u run_mlffr.py -b xdp_fwd -d xdp_fwd_test -v o2 -n 1 -c 6
```

## Step 7 — Generate graphs and read results

On **node-1**:

```bash
cd /usr/local/trex-configuration/visualize-data-scripts
python3 generate_graphs.py -d ~/xdp_fwd_<type>_all -b xdp_fwd -r 3
```

Output:
- `~/xdp_fwd_<type>_all/avgL/avg.png` — average latency vs offered load, all variants
- `~/xdp_fwd_<type>_all/rx/avg.png` — throughput (Rx rate) vs offered load, all variants

Copy to your local machine:
```bash
scp <user>@<node-1>:~/xdp_fwd_<type>_all/avgL/avg.png .
scp <user>@<node-1>:~/xdp_fwd_<type>_all/rx/avg.png .
```

**Reading the results:**

The MLFFR (Maximum Loss-Free Forwarding Rate) is the highest Tx rate at which the Rx
rate still tracks Tx linearly — the point just before the throughput curve bends flat.
Read the latency at that same Tx rate from the latency graph: that is the tail latency.

The claim to validate: **K2 variants (especially `k2`) have lower tail latency than
clang `o2` at the same or better MLFFR.**

Raw per-iteration data is in `~/xdp_fwd_<type>_all/MLFFR_<variant>_avgL.txt`
(format: `Rate,iter0,iter1,iter2`). Valid latency at the MLFFR is 30–80 µs. Values
above 500 µs indicate XDP is not active — see troubleshooting below.

## After a reboot

The static ARP entry on node-0 does not survive reboots. Re-add it before running again:

```bash
# On node-0:
sudo arp -s 10.10.1.2 <node-1-nic-mac>
```

The compiled files in `/usr/local/throughput-experiments/` survive reboots. `/tmp/sigcomm21_artifact` does not — re-clone if you need to modify configs:

```bash
git clone --branch hardware-generalization \
    git@github.com:MaryAyobami/sigcomm21_artifact.git /tmp/sigcomm21_artifact
```

## Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| avgL > 500 µs at all rates | XDP not active; packets in kernel slow path | Check `arp -n` on node-0 for 10.10.1.2 (must not be `incomplete`); re-run `sudo arp -s 10.10.1.2 <mac>` |
| `ip -d link show` shows `xdpgeneric` | NIC firmware or kernel does not support native XDP | Update NIC firmware; for c6620 ensure HWE kernel is installed |
| `failed to find BTF for extern` in dmesg | Stale or wrong `.o` file loaded | Re-run `setup_xdp_node0.sh` |
| `ERROR encountered while configuring TRex` | Stale `t-rex-64` process holding ZMQ port | `sudo pkill t-rex-64` then retry |
| setup.sh fails: `Missing trex_cfg_<type>.yaml` | Config file not created yet | Complete Step 3 first |
| k0–k4 results identical to o2 | Old placeholder objects in place | Re-run `setup_xdp_node0.sh` — it now installs real K2 objects from `k2-objects/` |
| c6420 (i40e): XDP loads but latency is high | RSS spreading traffic across queues | `sudo ethtool -X <iface> equal 1` on node-0 to restrict to one RX queue |
